# Secret Manager Service — Architecture Design

> Proto package: `secret.v1`
> Proto file: `proto/secret/v1/secret.proto`

## Overview

The Secret Manager is a **global service** (not retask-specific) that securely stores and retrieves secret values. It is
designed as a strict, BE-BE-only service — end-user tokens never reach it. The sole consumer is
the [Sandbox Service](./retask-sandbox.md):

- **Write path**: creates/updates/deletes secrets and caches masked values for display.
- **Read path**: calls `AccessSecrets` inside `GetSessionRuntime` to materialize the proxy's runtime view of a session. The sandbox proxy never calls secret-manager directly — it goes through the sandbox service.

## Architecture

```
┌──────────────────┐          ┌──────────────────┐
│  Sandbox Service │──BE-BE──▶│  Secret Manager  │
│                  │          │  (secret/v1)     │
│  SetSecret       │          │                  │
│  GetSecret       │          │  Stores secrets  │
│  DeleteSecret    │          │  encrypted       │
│  CopySecret      │          │                  │
│  AccessSecrets   │          │                  │
│  (runtime for    │          │                  │
│   GetSessionRun- │          │                  │
│   time + fork)   │          │                  │
└──────────────────┘          └──────────────────┘
```

### Key Properties

- **BE-BE only.** All RPCs reject user tokens. Only backend services with service-account auth can call secret-manager.
- **No user-facing UI.** Secrets are managed indirectly via sandbox/template config forms. There is no standalone "
  Secrets" management page.
- **No soft delete.** `DeleteSecret` performs immediate hard-delete. No grace period needed (
  see [Deletion Safety](#deletion-safety)).
- **No outbound calls.** Secret-manager is a pure storage service. It publishes no events.
- **Workspace-scoped.** Every secret belongs to a workspace (`workspace_id` is required).

## Entity: Secret

```protobuf
message Secret {
  string                    secret_id = 1;   // UUID
  string                    workspace_id = 2;   // required
  .common.v1.Nrn            related_to_nrn = 3;   // owning resource NRN (sandbox or template)
  string                    name = 4;   // user-facing label, e.g. "ANTHROPIC_API_KEY"
  string                    description = 5;
  string                    masked_value = 6;   // e.g. "sk-•••••c123"
  // write-only: accepted on SetSecret, never stored as plaintext.
  // never returned by GetSecret (only by AccessSecrets)
  string                    value = 7;
  bytes                     encrypted_value = 12;
  // Audit (no soft delete — hard delete only)
  .common.v1.Nrn            created_by_nrn = 8;
  google.protobuf.Timestamp created_at = 9;
  .common.v1.Nrn            updated_by_nrn = 10;
  google.protobuf.Timestamp updated_at = 11;
}
```

### Field Notes

- **`secret_id`** — UUID, the `resource_id` portion of the secret's NRN (`nweb:secret:secret:<uuid>`).
- **`related_to_nrn`** — points to the resource that owns this secret (e.g., `nweb:retask-sandbox:sandbox:<uuid>` or
  `nweb:retask-sandbox:sandbox-template:<uuid>`). Used for cascade deletion. One sandbox/template can have many secrets,
  so `related_to_nrn` is NOT a primary key.
- **`value`** — the plaintext secret. Accepted on `SetSecret` (write-only, `bson:"-"`). Never stored directly —
  encrypted into `encrypted_value` before persistence. Never returned by `GetSecret`. Only returned by `AccessSecrets`.
- **`encrypted_value`** — the value encrypted with the workspace's DEK. Stored in MongoDB. Never returned to callers —
  only used internally for decryption.
- **`masked_value`** — generated by secret-manager on write (e.g., show first 3 + last 4 chars). Returned by `GetSecret`
  for display caching.

## Entity: EncryptionKey

Internal entity (no RPCs expose it). Stores the per-workspace DEK wrapped by GCP KMS.

```protobuf
message EncryptionKey {
  string                    workspace_id = 1;   // also the _id
  bytes                     wrapped_dek = 2;   // DEK encrypted by KMS KEK
  google.protobuf.Timestamp created_at = 3;
}
```

### Encryption Model (Envelope Encryption)

Secret values are encrypted using **envelope encryption** with per-workspace isolation:

```
┌─────────────┐         ┌──────────────┐
│  GCP KMS    │         │  MongoDB     │
│             │         │              │
│  KEK        │◀─wrap──▶│  EncryptionKey│
│  (master)   │  unwrap │  .wrapped_dek│
│             │         │              │
│  never      │         │  Secret      │
│  leaves KMS │         │  .encrypted_ │
│             │         │   value      │
└─────────────┘         └──────────────┘
```

| Layer                         | What                                               | Where                                                |
|-------------------------------|----------------------------------------------------|------------------------------------------------------|
| **KEK** (Key Encryption Key)  | Master key, encrypts/decrypts DEKs                 | GCP KMS (never leaves)                               |
| **DEK** (Data Encryption Key) | Per-workspace key, encrypts/decrypts secret values | MongoDB `encryption_keys` collection, wrapped by KEK |
| **Secret value**              | The actual secret data                             | MongoDB `secrets` collection, encrypted by DEK       |

### Key Lifecycle

- **First secret in a workspace** → generate a random AES-256 DEK, wrap it with KMS, store as `EncryptionKey`
- **SetSecret** → look up `EncryptionKey` by `workspace_id`, unwrap DEK via KMS, encrypt `value` → `encrypted_value`
- **AccessSecrets** → same lookup, unwrap DEK, decrypt `encrypted_value` → `value`
- **Workspace deleted** → delete the `EncryptionKey` row → all secrets become unrecoverable

## RPC Surface (6 RPCs)

```protobuf
service SecretService {
  // Create or update a secret. Accepts plaintext `value`, stores encrypted.
  // Returns the secret ID.
  rpc SetSecret(Secret) returns (.common.v1.Id) {
    option (google.api.http) = {post: "/v1/secrets" body: "*"};
  }

  // Get a secret's metadata (name, description, masked_value).
  // Never returns plaintext value.
  rpc GetSecret(.common.v1.Id) returns (Secret) {
    option (google.api.http) = {get: "/v1/secrets/{id}"};
  }

  // Batch-read secret metadata for known IDs within a workspace.
  // Both filter.workspace_id and filter.secret_ids are required —
  // there is no enumerate-all mode. Never returns plaintext value.
  rpc GetSecrets(SecretsRequest) returns (SecretsResponse) {
    option (google.api.http) = {get: "/v1/secrets"};
  }

  // Batch-read plaintext secret values. BE-BE only, called by sandbox service.
  // Used for runtime resolution inside GetSessionRuntime and for template fork.
  rpc AccessSecrets(AccessSecretsRequest) returns (AccessSecretsResponse) {
    option (google.api.http) = {post: "/v1/secrets:access" body: "*"};
  }

  // Immediate hard-delete. No soft delete, no grace period.
  rpc DeleteSecret(.common.v1.Id) returns (.common.v1.Empty) {
    option (google.api.http) = {delete: "/v1/secrets/{id}"};
  }

  // Copy an existing secret to a new owning resource within the same workspace.
  // Plaintext is re-encrypted internally — never crosses the process boundary.
  // Returns the new Secret with `value` stripped (metadata only).
  rpc CopySecret(CopySecretRequest) returns (Secret) {
    option (google.api.http) = {post: "/v1/secrets/{secret_id}:copy" body: "*"};
  }
}
```

### GetSecrets (Batch Metadata Read)

```protobuf
message SecretsRequest {
  message Filter {
    string          workspace_id = 1;   // required
    repeated string secret_ids   = 2;   // required — caller must already know IDs
    .common.v1.Nrn  related_to_nrn = 3; // optional — further restrict to a single owning resource
  }
  Filter filter = 1;
}

message SecretsResponse {
  repeated Secret secrets = 1;   // metadata only, no plaintext value
}
```

`GetSecrets` is a batched form of `GetSecret`: it returns the same metadata (`name`, `description`, `masked_value`, audit fields) for many IDs in one call, scoped to a workspace. It does **not** return plaintext — that remains exclusive to `AccessSecrets`.

Both `workspace_id` and `secret_ids` are required. The RPC rejects requests where either is empty. This preserves the non-enumerable property: a caller that does not already hold the secret IDs (typically stored alongside `secret_nrn` on the owning resource) cannot use this RPC to discover them. There is no list-all mode.

`related_to_nrn` is optional. When set, results are further restricted to secrets owned by that resource NRN — useful for callers that want to confirm the returned secrets all belong to a specific sandbox or template. It is a filter on top of `secret_ids`, not a replacement for it.

The primary use case is the sandbox service hydrating cached `masked_value` for many secrets at once when rendering a config form, replacing N sequential `GetSecret` calls with one round trip.

### CopySecret

```protobuf
message CopySecretRequest {
  string         workspace_id          = 1;   // required
  string         secret_id             = 2;   // required — source secret to copy
  .common.v1.Nrn source_related_to_nrn = 3;   // required — expected owner of source
  .common.v1.Nrn target_related_to_nrn = 4;   // required — owner for the new copy
}
```

`CopySecret` duplicates a single existing secret under a new owning resource within the same workspace. The primary use case is forking a sandbox from a template: each template-owned secret is copied so the new sandbox owns its own independent copy, and subsequent edits or deletions on either side don't affect the other.

The handler reuses the existing `GetSecret` handler to load the source (inheriting its permission checks), then enforces two extra equality checks before copying:

1. `loaded.workspace_id == request.workspace_id`
2. `loaded.related_to_nrn == request.source_related_to_nrn`

Either mismatch returns `InvalidArgument`. The `source_related_to_nrn` parameter prevents a caller that happens to know a `secret_id` (e.g., from a leaked log) from copying a secret out of a resource it has no business reading.

The plaintext is decrypted and re-encrypted entirely inside secret-manager — it never crosses a process boundary, unlike the alternative of `AccessSecrets` + `SetSecret`. The new copy gets a fresh `secret_id`, fresh ciphertext bytes, fresh audit fields, and `related_to_nrn = target_related_to_nrn`. `name`, `description`, and `masked_value` are copied from the source. The response is the new `Secret` with `value` stripped (metadata only).

There is no batch form yet — callers loop if they need to copy multiple secrets. Cross-workspace copy is not supported (DEKs are per-workspace).

### AccessSecrets (Batch Read)

```protobuf
message AccessSecretsRequest {
  repeated string secret_ids = 1;
}

message AccessSecretsResponse {
  repeated SecretAccess secrets = 1;
}

message SecretAccess {
  string secret_id = 1;
  string value = 2;   // plaintext
}
```

### No GetSecrets (List)

There is no list/search endpoint. Secrets are managed indirectly through sandbox/template configs. The sandbox service
knows which `secret_nrn` values it stores and fetches them individually via `GetSecret` or in batch via `AccessSecrets`.

## Data Flows

### Creating a Secret (via Sandbox Config)

```
User types "sk-abc123" into a secret env var field in FE
  → FE sends SetSandbox with Config.EnvVar.SecretValue.value = "sk-abc123"
  → Sandbox service calls secret-manager.SetSecret(Secret{
      workspace_id: "...",
      related_to_nrn: sandbox NRN,
      name: env var key,
      description: env var description,
      value: "sk-abc123"
    })
  → Secret-manager:
      1. Generates masked_value: "sk-•••••c123"
      2. Encrypts value and stores
      3. Returns Id
  → Sandbox service stores secret_nrn + masked_value on EnvVar
  → Sandbox service strips SecretValue.value (bson:"-")
```

### Reading a Secret (Display)

```
FE calls GetSandbox
  → Sandbox service returns config with SecretValue{secret_nrn, masked_value, value: ""}
  → FE displays masked_value directly — no call to secret-manager needed
```

### Accessing a Secret (Runtime Injection)

```
Proxy starts a session, needs real env var values
  → Proxy calls sandbox.GetSessionRuntime(session_id)     (BE-BE, service token)
  → Sandbox service:
      1. Loads Session + Sandbox (rejects if either is soft-deleted)
      2. Collects secret_ids from Sandbox.Config.env_vars
      3. Calls secret-manager.AccessSecrets(workspace_id, secret_ids)
      4. Re-injects plaintext into each EnvVar.SecretValue.value
         (leaving `value=""` for any secret that can't be resolved)
      5. Returns SessionRuntime{session, sandbox}
  → Proxy injects env vars into the sandbox shell from the returned Config
```

The proxy no longer talks to secret-manager directly. Sandbox service is the single gateway, so all authorization, audit (via `SandboxActivity`), and partial-failure handling for runtime secret access live in one place.

### Updating a Secret

```
User changes a secret env var value in FE
  → Sandbox service diffs old vs new config
  → For the changed key: calls secret-manager.SetSecret with the same related_to_nrn
  → Secret-manager updates encrypted value + masked_value
  → Sandbox service updates cached masked_value
```

### Forking a Template's Secrets (Sandbox Fork)

```
User forks a sandbox from a template
  → Sandbox service reads template config: collects (template_secret_nrn, env var key) pairs
  → For each pair, sandbox service calls secret-manager.CopySecret(
      workspace_id: "...",
      secret_id: template_secret_id,
      source_related_to_nrn: template NRN,
      target_related_to_nrn: new sandbox NRN
    )
  → Secret-manager:
      1. Loads source via GetSecret (perm checks)
      2. Verifies workspace_id and source_related_to_nrn match
      3. Re-encrypts value under workspace DEK
      4. Inserts new Secret with new id + target NRN, returns metadata
  → Sandbox service stores the new secret_nrn + masked_value on the new sandbox's EnvVar
```

### Deleting a Secret (via Config Update)

```
User removes a secret env var from config
  → Sandbox service diffs old vs new config
  → For the removed key: calls secret-manager.DeleteSecret(old_secret_nrn)
  → Secret-manager hard-deletes immediately
```

## Deletion Safety

Secrets use **immediate hard-delete**, not soft-delete. This is safe because:

1. The proxy resolves secrets at session start and holds values in memory (injected as shell env vars).
2. When a user removes a secret from a config, the old `secret_nrn` disappears from the config. No code path will ever
   try to resolve it again.
3. Running sessions continue unaffected — they already have the values in memory.
4. The timing window between `DeleteSecret` and the proxy's next tick is ~60s, but the proxy reads the new config (which
   no longer references the deleted secret), so the deleted secret is never accessed.

## Cascade Deletion (Event Handlers)

Secret-manager listens to three events for cascade cleanup:

### WorkspaceDeleted (from workspace service)

Hard-delete all Secrets where `workspace_id` matches the deleted workspace. This is a belt-and-suspenders catch-all —
individual sandbox/template deletions also trigger cleanup.

### SandboxDeleted (from sandbox service)

Hard-delete all Secrets where `related_to_nrn` matches the deleted sandbox NRN (`nweb:retask:sandbox:<uuid>`).

### SandboxTemplateDeleted (from sandbox service)

Hard-delete all Secrets where `related_to_nrn` matches the deleted template NRN (`nweb:retask:sandbox-template:<uuid>`).

### Cascade Order

When a workspace is deleted:

1. Sandbox service soft-deletes templates → publishes `SandboxTemplateDeleted` → secret-manager cleans up template
   secrets
2. Sandbox service soft-deletes sandboxes → publishes `SandboxDeleted` → secret-manager cleans up sandbox secrets
3. Secret-manager also receives `WorkspaceDeleted` directly → catches any orphans

## Implementation Notes for Go Service

- **Envelope encryption.** Use GCP KMS as the KEK provider. Generate one AES-256 DEK per workspace, wrap it with KMS,
  and store the wrapped DEK in the `encryption_keys` collection. Encrypt `Secret.value` with the workspace's DEK and
  store the ciphertext in `Secret.encrypted_value`. The plaintext `value` field (`bson:"-"`) is never persisted.
- **`value` field in responses.** `GetSecret` must strip the `value` field from its response. Only `AccessSecrets`
  returns plaintext values.
- **`masked_value` generation.** Generated server-side on `SetSecret`. Suggested format: first 3 chars + "•••••" + last
  4 chars. If the value is too short (< 8 chars), mask entirely as "••••••••".
- **Auth enforcement.** All RPCs require service-account auth. The service must reject any request with a user token.
  This is the primary security boundary — even if a user somehow discovers a `secret_id`, they cannot call `GetSecret`
  or `AccessSecrets` directly.
- **`GetSecrets` is not a list RPC.** It is a batched lookup that requires both `workspace_id` and `secret_ids`. Reject any request with either field empty — there is no enumerate-all mode. The only ways to
  learn a secret ID remain: (a) you stored its `secret_nrn` in your own entity, or (b) you receive it via cascade events
  through `related_to_nrn`.Also enforce that every returned secret's `workspace_id` matches the request — a caller from workspace A must not be able to read a secret from workspace B by guessing IDs.
- **`related_to_nrn` indexing.** Create a MongoDB index on `related_to_nrn` for efficient cascade deletion queries. Also
  index `workspace_id` for the `WorkspaceDeleted` cascade.
- **`encryption_keys` collection.** Store `EncryptionKey` documents keyed by `workspace_id`. On `WorkspaceDeleted`,
  delete the corresponding `EncryptionKey` to make all workspace secrets unrecoverable.
